# PRD 메타 프롬프트 — Notion 견적서 웹 뷰어 MVP

이 파일은 LLM(Claude / GPT 등)에게 **MVP PRD를 생성하라고 시키는 프롬프트
템플릿**입니다. 본문은 두 영역으로 나뉘어 있습니다.

- **사용법** (지금 이 영역) — 사람이 읽는 안내. LLM에 붙여넣지 마세요.
- **프롬프트 본문** — `===== LLM에 복사할 구간 시작 =====` 줄부터
  `===== LLM에 복사할 구간 끝 =====` 까지. **이 사이를 통째로 복사**해서
  LLM 채팅에 붙여넣으면 MVP PRD가 출력됩니다.

## 사용법

1. 아래 "프롬프트 본문" 구간 전체를 복사한다.
2. Claude 또는 ChatGPT(긴 컨텍스트 모델 권장)에 그대로 붙여넣는다.
3. LLM이 PRD를 한국어로 출력하면, 그대로 또는 약간 다듬어 `docs/PRD.md`
   같은 파일로 저장한다.
4. 결정사항을 바꾸고 싶다면 **블록 3(확정 아키텍처)** 의 표만 수정한 뒤
   다시 돌린다. 다른 기능의 PRD를 만들고 싶다면 **블록 2(컨텍스트)**·
   **블록 3(결정)** 을 통째로 교체한다.

> 메타 팁: LLM이 PRD를 너무 길게/짧게 내거나, 결정사항을 재논의하려고 하면
> 블록 4(출력 스키마) 또는 블록 5(제약)에 더 강한 문구를 추가하세요.

# [블록 1] 역할

당신은 B2B SaaS 스타터 프로젝트의 **시니어 프로덕트 매니저**입니다.
당신의 임무는 엔지니어가 즉시 티켓으로 쪼개 구현에 착수할 수 있도록,
실행 가능한 **MVP PRD를 한국어 마크다운**으로 작성하는 것입니다.

영업·마케팅 톤이 아니라 **엔지니어링 PRD** 입니다. 시장 규모·매출
시뮬레이션·페르소나 심리 묘사 같은 마케팅 요소는 쓰지 마세요. 무엇을
만들지·어떻게 만들지·어디까지 만들지에만 집중하세요.

# [블록 2] 컨텍스트

## 제품 한 줄 정의

> Notion 데이터베이스에 입력된 견적서를, 토큰이 포함된 비밀 URL로
> 클라이언트에게 공유하여 **웹에서 열람**하고 **PDF로 다운로드**받게
> 하는 도구.

## 사용자 페르소나

- **(A) 발행자** — 1인 사업자 / 소규모 에이전시 대표. Notion으로 업무를
  관리하며, 견적서도 Notion DB row로 작성한다. 별도 견적서 SaaS를 쓰는
  마찰을 피하고 싶다.
- **(B) 수신자(클라이언트)** — 견적을 의뢰한 사람. 회원가입 없이 받은
  링크로 웹에서 견적을 확인하고, 필요하면 PDF로 저장해 사내 결재에
  올리고 싶다.

## 기술 스택 고정값 (이미 결정됨)

다음은 코드베이스에 이미 반영된 사실들입니다. PRD는 이들과 충돌하면
안 됩니다.

- **프레임워크**: Next.js 15 App Router, React 19
- **Next 15 비동기 API**: `params`, `searchParams`, `cookies()`,
  `headers()` 는 모두 `Promise`. 페이지/라우트 핸들러는 `async` 로
  선언하고 `await` 해야 함.
- **렌더링 기본값**: 모든 페이지/레이아웃은 Server Component. 클라이언트
  컴포넌트는 인터랙션이 필요한 작은 섬(island) 으로 한정.
- **fetch 캐싱**: Next 15 의 `fetch()` 기본은 `no-store`. 캐싱이
  필요하면 `{ cache: 'force-cache' }` 또는 `{ next: { revalidate: N } }`
  을 명시.
- **UI**: shadcn `radix-nova` 스타일. `components/ui/*` 의 button,
  card, input, label 등을 우선 재사용. 새 컴포넌트가 필요하면
  `npx shadcn@latest add <name>` 로 추가.
- **스타일링**: Tailwind v4, OKLCH 토큰 기반 (`app/globals.css`). 다크
  모드는 `next-themes` 가 `<html>` 의 `.dark` 클래스를 토글.
- **사이트 메타 단일 소스**: `lib/site-config.ts` 의 `siteConfig` 객체.
  새 네비게이션 항목은 반드시 이 파일에만 추가.
- **에러 처리**: `app/error.tsx` 는 `{ error, reset }` 시그니처를 받는
  `"use client"` 컴포넌트.
- **개발 서버**: `npm run dev` (반드시 `--turbopack` 플래그 유지).

## 절대 제안하지 말 것

- Pages Router, `getServerSideProps`, `getStaticProps`
- `params`/`searchParams` 를 동기로 가정하는 코드
- `fetch()` 의 기본 캐싱이 `force-cache` 라고 가정
- 자체 다크모드 토글 / 자체 폰트 변수 / 별도 CSS 변수 시스템
- 클라이언트 사이드 PDF 라이브러리 (html2canvas + jsPDF 등)
- 견적서 화면을 클라이언트 컴포넌트로 전부 감싸는 설계

# [블록 4] 출력 스키마

PRD 는 정확히 다음 9개 섹션을, **이 순서로**, **이 헤더로** 출력하세요.
섹션 누락·순서 변경·헤더 변경 금지.

## 1. 제품 개요

- 3문장 이내. 한 줄 정의 + 핵심 가치 + MVP 범위 한 줄.

## 2. 타깃 사용자 & 핵심 시나리오

- 페르소나 A(발행자), B(수신자) 각각에 대해 1-2개의 시나리오를
  "사용자가 …할 때 → 시스템이 …한다" 형태로.

## 3. MVP 범위 (In / Out)

- 표 형식. 좌측 컬럼 "포함", 우측 컬럼 "제외". 각 8-15개 항목.

## 4. 확정 아키텍처 결정

- 블록 3 의 표를 그대로 옮기되, 각 결정에 한 줄 보충 설명을 추가해도
  좋음. 단 결정 자체는 변경 금지.

## 5. 데이터 모델

- **Notion DB 스키마**: 표 형식. 컬럼: `필드명 / Notion 타입 / 예시값 /
필수 여부`. 최소 필드: 견적서 번호, 클라이언트명, 발행일, 만료일,
  항목 배열(이름/수량/단가), 합계, 부가세, 총합계, 메모, 접근 토큰,
  상태(draft/sent/viewed).
- **앱 내부 타입**: TypeScript 타입 시그니처. `Invoice`, `InvoiceItem`,
  `InvoiceStatus` 정도. 구현 코드는 쓰지 말고 시그니처만.

## 6. 라우팅 & 화면

다음 3개 경로를 각각 한 문단 + 시그니처로 설명:

- `app/invoice/[id]/page.tsx` — Server Component. `params`·
  `searchParams` 모두 Promise 이므로 await. 토큰 검증 → Notion fetch →
  렌더. 토큰 불일치 시 404 또는 전용 에러 화면.
- `app/api/invoice/[id]/pdf/route.ts` — Route Handler. 동일한 토큰
  검증 후 PDF 스트림 응답 (`Content-Type: application/pdf`,
  `Content-Disposition: attachment; filename="..."`).
- `app/invoice/[id]/error.tsx` — `"use client"`. `{ error, reset }`
  시그니처. 토큰 오류 / Notion 404 / 외부 API 장애를 사용자 친화 문구로
  안내.

## 7. 보안 모델

- 토큰 생성 규칙 (예: `crypto.randomBytes(32).toString('base64url')`).
- 토큰 저장 위치: Notion DB 의 별도 필드 vs 앱 측 별도 store — 선택과
  이유.
- 환경 변수: `NOTION_TOKEN`, `NOTION_DATABASE_ID` 는 **서버 전용**.
  `NEXT_PUBLIC_` 접두사 금지.
- 토큰 비교는 **상수 시간 비교**(`timingSafeEqual`).
- 견적서 응답 헤더: `Cache-Control: no-store`, `X-Robots-Tag: noindex`.

## 8. 마일스톤 & 작업 분해

- 5-8 개 체크포인트. 각 항목 양식:
  - **제목**
  - **무엇을 한다** (1-3 문장)
  - **Done = …** (검증 가능한 완료 기준 1-3 개)
- 권장 분해 예시(참고용, 실제로는 PRD 가 판단):
  1. Notion DB 스키마 정의 + 더미 row 생성
  2. `@notionhq/client` 설치 + 서버 전용 fetch 유틸 (`lib/notion.ts`)
  3. `app/invoice/[id]/page.tsx` 라우트 + 토큰 검증
  4. 견적서 뷰 UI (shadcn `Card`, `Table` 기반)
  5. `@react-pdf/renderer` 설치 + PDF 컴포넌트
  6. `app/api/invoice/[id]/pdf/route.ts` 스트리밍
  7. `error.tsx` + 404 / 토큰 실패 UX

## 9. 검증 & 수용 기준

- **수동 검증 시나리오**:
  - 올바른 `?token=...` 으로 견적서 열람 성공
  - 잘못된/누락 `token` 으로 접근 시 401 또는 404 화면
  - "PDF 다운로드" 클릭 시 PDF 파일이 정상 다운로드되고 한글이 깨지지
    않음
  - Notion 에서 견적서 row 를 수정하고 새로고침하면 변경분이 반영됨
- **자동 검증**:
  - `npm run build` 성공, 정적 분석 경고 없음
  - `npm run lint` 무경고
  - `/invoice/[id]` 페이지가 빌드 로그에서 동적 렌더링으로 표시되는지
    확인 (SSG 가 아님)

# [블록 5] 제약 & 금지 사항

- **분량**: 전체 PRD 250-400 줄. 초과 시 섹션 8(마일스톤)·9(검증) 의
  중복을 우선 줄이세요.
- **머리말·맺음말 금지**: "이 PRD 는 ..." / "이상으로 PRD 를 마칩니다"
  같은 문장 금지. 곧장 섹션 1 로 시작.
- **가상의 통계 금지**: "시장의 80% 가 ..." / "평균 견적 발행 시간이
  3분 단축" 같은 추정치 금지. 근거 있는 정성적 서술만.
- **구현 코드 금지**: 함수 구현·React JSX 본문·SQL 쿼리 본문은 쓰지
  마세요. 타입 시그니처·라우트 시그니처·환경 변수 이름·헤더 키만.
- **선택지 재제시 금지**: 블록 3 의 4 가지 결정은 PRD 내에서 다시
  "옵션 A vs B" 형식으로 비교하지 마세요. 결정과 근거만 적습니다.
- **"추후 고려" 섹션**: 최대 1개까지 허용. 권장 항목: 결제 연동,
  이메일 발송, 다국어, 알림, 견적서 상태 변경 워크플로.
- **출력 형식**: 순수 마크다운. 인사말·이모지 없이.
